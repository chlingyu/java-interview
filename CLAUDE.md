# CLAUDE.md - 项目约束

## 八股文面试题写作风格规范（B+C 混合模式）

### 目标读者
基础较弱的 Java 开发者，准备 3~5 年经验的后端面试。假设读者只有"能写 CRUD"的水平，对底层原理了解很少。

### 写作模式：B+C 混合

根据题目的面试高频程度，使用不同深度的写作风格：

#### ⭐⭐⭐⭐~⭐⭐⭐⭐⭐ 高频题 → C 风格（通俗故事型）

1. **用生活比喻解释原理**：先用一个通俗场景让读者理解概念，再过渡到技术细节
2. **比喻和技术之间要有过渡段**：比喻讲完后，不要直接甩表格/代码。用 2~3 句话把比喻中的角色"翻译"成技术概念，让读者带着理解往下读。例如："上面说的'格子'就是 HashMap 的数组槽位，'手机尾号'就是 hashCode 对数组长度取模的结果……"
3. **专业术语首次出现时标注**：用 **加粗（一句话解释）** 格式。括号里的解释必须是人话，不能用另一个术语解释术语。
   - ✅ 好的：**STW（Stop The World，所有业务线程暂停，就像全公司停工等保洁打扫完）**
   - ❌ 坏的：**栈帧（局部变量表、操作数栈、动态链接、返回地址）** ← 四个生词解释一个生词，等于没解释
4. **关键数字用 ⚡ 标注**：所有阈值、默认值必须写死，不要模糊表述。例如：链表长度 ⚡**≥ 8** 且数组长度 ⚡**≥ 64** → 转红黑树
5. **信息密度控制**：一个知识点内，表格行数不超过 5~6 行。如果对比项超过 6 个，拆成"核心对比"和"补充了解"两部分，核心部分重点讲 2~3 个最常考的，其余简要带过
6. **复杂流程用分步骤 + 示意图**：拆成带编号的步骤，每步用一句通俗话概括"这步在干嘛"，再跟技术细节
7. **每题附"🎤 面试这样答"**：给出一段可以直接参考的面试回答话术
8. **配图要求**：优先使用 Mermaid 语法画流程图/结构图（GitHub 原生渲染），复杂数据结构可用 ASCII 图。避免纯文字描述空间关系

#### ⭐⭐~⭐⭐⭐ 普通题 → B 风格（结构化讲解型）

1. **一句话回答** + 关键原理简述
2. **加一句通俗过渡**：不需要完整比喻，但要有 1~2 句"说人话"的解释帮助理解，不能上来就是纯术语表格。例如："简单说就是 XXX，你可以理解为 XXX"
3. 保留专业术语标注（同 C 风格的标注规则，术语解释必须是人话）
4. 核心代码示例（带中文注释）
5. 用表格对比差异，简洁明了
6. 不需要完整比喻故事和"🎤 面试这样答"

### 每道题的结构模板

#### 高频题（C 风格）模板

```markdown
### N. 面试题目？

> ⭐⭐⭐⭐⭐ 必考 | 难度：⭐⭐⭐

**一句话回答**：xxxxx

**通俗理解**：（用比喻/生活场景解释原理）

**回到技术**：（2~3 句过渡，把比喻"翻译"成技术概念）

**原理详解**：（分步骤 + 关键数字 ⚡ 标注 + Mermaid/ASCII 图）

**代码示例**：（核心代码，带中文注释）

**🎤 面试这样答**：
> "xxxxxx"
```

#### 普通题（B 风格）模板

```markdown
### N. 面试题目？

> ⭐⭐ 了解 | 难度：⭐⭐

**回答**：一句话回答 + 1~2 句通俗过渡。然后再展开讲解原理，专业术语加粗标注。

**代码/表格**：（如需要）
```

### 注意事项

- **不要写干巴巴的纯表格堆砌**，高频题必须有温度
- **不要省略关键数字**，所有阈值、默认值都要写清楚
- **专业术语不能落下**，但第一次出现必须用通俗语言解释，不能用术语解释术语
- **代码示例要有中文注释**，不要写纯英文注释
- **篇幅控制**：高频题可以长，低频题要精简，每章控制在合理长度内
- **比喻到技术不能断层**：写完比喻后必须有过渡段，把比喻映射回技术概念，再进入详解
- **表格不能当万能工具**：超过 6 行的对比表格要拆分，核心的重点讲、次要的简要带过
- **B 风格也要有人味**：即使是低频题，也要有 1~2 句通俗解释，不能直接甩术语+表格
